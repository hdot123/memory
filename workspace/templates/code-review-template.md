# 代码审查任务

## 0. 全局规则（所有审查员必须遵守）

### 严重度标尺
- **P0** — 必崩 / 数据丢失 / 安全漏洞 / 核心功能完全失效
- **P1** — 功能结果错误 / 状态不一致 / 运行时异常未捕获 / 静默失败（吞掉异常但假装正常）
- **P2** — 边界情况崩溃 / 配置异常导致降级 / 竞态条件 / 路径逃逸
- **P3** — 误导性错误消息 / 日志缺失 / 验证不完整 / 代码异味

### 报告格式（每个 bug 必须包含全部 5 项，缺一不计入最终报告）
1. **文件绝对路径 + 精确行号**
2. **严重度 + 分类**（崩溃 / 逻辑错误 / 边界 / 安全 / 竞态 / 静默失败）
3. **触发条件**（具体的输入/配置/状态，禁止写"理论上可能"）
4. **崩溃链路**：入口 → 调用链 → 出错点的完整路径（伪代码形式）
5. **预期行为 vs 实际行为**（两句话，直接对比）

### "非发现"举证要求
声称某段代码没问题的，必须写明：
- 验证了哪些字段/方法/路径
- 从哪些调用方交叉验证过
- 具体验证了什么（如"逐条比对了构造函数参数与 from_gateway_kwargs 传入的字段"）
禁止只写"未发现问题"或"I did not find a concrete issue"跳过。

### 发现者标注
每个 bug 末尾标注发现者编号（如 `R3, R7`），多人独立发现同一问题时全部列出。

## 1. 审查员分配

> 以下由前置分析脚本自动生成，手动调整后分发。

### 第一层：主审员（逐行审查）

| 编号 | 主文件 | 行数 | 专项维度 |
|------|--------|------|----------|
| R1   | ...    | ...  | ...      |
| R2   | ...    | ...  | ...      |
| ...  | ...    | ...  | ...      |

**规则**：每人对自己主文件做完整逐行审查，同时覆盖分配的专项维度。

### 第二层：交叉审查员（接口边界）

| 编号 | 交叉方向 | 覆盖接口文件 |
|------|----------|--------------|
| R6   | ... → ... | ...          |
| R7   | ... → ... | ...          |
| ...  | ...      | ...          |

**规则**：
- 不重复逐行审查，只看跨文件接口边界
- 重点：参数是否匹配、返回值结构是否一致、异常是否能传播到调用方
- 特别关注：方法名/参数名是否完全一致（大小写、下划线、拼写）

### 第三层：专项审查员（全模块扫描）

| 编号 | 专项方向 |
|------|----------|
| R9   | 安全 + 并发 |
| R10  | 崩溃链路 |
| ...  | ...      |

**安全+并发专项检查清单**：
- [ ] 路径遍历（所有 Path 拼接、_path_is_under、symlink）
- [ ] TOCTOU（所有 exists + read/write 序列）
- [ ] 锁泄漏（flock/mutex 的异常路径）
- [ ] 输入验证（环境变量、配置键缺失、空值）
- [ ] 文件操作原子性（partial write、temp rename）
- [ ] 并发写入（多进程/多线程写同一文件）
- [ ] 代码注入（eval/exec/__import__/动态加载）
- [ ] 模块级状态污染（globals/单例/类变量）

**崩溃链路专项检查清单**：
- [ ] 从每个公开入口出发，追踪每条执行路径
- [ ] 逐个标注：哪里可能抛异常、上层是否捕获、未捕获的后果
- [ ] 特别关注 except 块内部本身可能抛异常的情况
- [ ] 静默失败：catch 后 return 空值/默认值但未记录日志

## 2. 执行步骤

```
Step 1 — 通读主文件，逐行标注可疑点
Step 2 — 对每个可疑点，构造具体触发条件
Step 3 — 追踪崩溃链路（从最近入口到出错点）
Step 4 — 从调用侧/被调用侧验证接口匹配（主审员必做）
Step 5 — 写入报告
```

---

## 3. 交叉验证规则

**主审员（R1-R5）**：
- 找到所有**调用你主文件**的模块 → 验证参数和返回值
- 找到你主文件**调用的所有外部模块** → 验证你传参是否正确

**交叉审查员（R6-R8）**：
- 只看跨文件接口边界，不重复逐行审查
- 逐方法比对：接口声明的方法名/参数 vs 实际实现的方法名/参数
- 重点检查：拼写差异、下划线差异、参数顺序差异

**专项审查员（R9-R10）**：
- 全模块扫描，不限文件
- 按维度清单逐项排查，不关注功能正确性

---

## 4. 禁止事项

- 禁止无具体触发条件的"理论可能"
- 禁止把明显崩溃降级为 P2（P0 标尺已明确）
- 禁止跳过交叉验证
- 禁止 Non-finding 只写一句话
- 禁止重复报告同一行同一问题的不同描述（发现者合并）

---

## 5. 输出格式

每人产出一份报告，文件名格式：`review/R{N}-{scope}-bugs.md`

最终合并规则：
1. 按"文件+行号"去重，同一行不同问题保留，重复问题合并
2. 严重度仲裁：同一 bug 不同人判了不同级别的，取更高的
3. 发现者合并：独立发现同一 bug 的审查员全部列出
4. 最终输出按 P0 → P3 排序
